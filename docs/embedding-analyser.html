Fragment analyser* — design & scoring logic

For each section of the approved sitemap, the analyser asks: does our prebuilt fragment library already cover this? The result feeds (a) the developer with a coverage % and a gap list to grow the library, and (b) the frontend agent with a deduped list of suggested fragments as a head start. The agent is otherwise fragment-blind during design, so the analyser is the bridge between the freely-designed mockup and the fragment-aware build. This doc captures the exact scoring logic so we can fine-tune it later. Sibling docs: understanding_v1.html (the whole pipeline) and design-pipeline.html (the plan).

0 The pipeline in one picture

Lazy compute, cache by sitemap hash. Each (page, section) becomes one embedding call; the closest fragment in vector space wins, two thresholds gate "good enough", and the result lands on disk + in the next frontend build's system prompt.

1 Trigger & cache lib/server/fragment-coverage.ts

Embeddings cost real money and add wall-clock latency, so the analyser never runs in the polling path. Two triggers ever invoke it: the Analyzer view (GET /api/apps/[id]/analyzer) and the frontend build turn (preponed into the system prompt). Both share one cache.

AspectDetail
Cache filedata/<appId>/design/coverage.json
Cache keySHA-256 of [{purpose, sections}] per page, JSON-stringified. Navigation, home, shellLayout, flows, durationMs, themes — all excluded (they don't affect matching).
Hit pathReturns cached FragmentCoverage instantly — zero embedding calls.
Miss pathLoops every (page, section), one embedding call each, writes the cache on completion.
Empty sitemapReturns null. UI shows "No sitemap to analyze yet — run the design stage first."
Design intent: the cache key is narrower than the file's hash. Changing a page's home flag or reordering navigation doesn't invalidate the analyser — only edits to a section's text or a page's purpose do.

2 Query construction

Each section is paired with the page-level context so the same section name on different pages becomes different queries.

// fragment-coverage.ts:100
const query = `${page.name}: ${section}. ${page.purpose}`;

Example queries built from a habit-tracker sitemap:

page.namesectionpage.purpose→ query
DashboardToday's habits listQuick check-in for active habits "Dashboard: Today's habits list. Quick check-in for active habits"
HabitsHabit detail headerBrowse and manage all habits "Habits: Habit detail header. Browse and manage all habits"

Why join the three fields? Section text alone ("List", "Header", "Filters") is too short to be discriminative; pairing it with page context anchors the embedding in the right semantic neighbourhood.

3 Embedding & index lib/server/fragment-index.ts

ConceptValue
Embedding modelopenai/text-embedding-3-small via OpenRouter
Dimensions1536
Indexsqlite-vec (vector_full_scan) — every fragment compared, no ANN approximation. Fragment library is small (~50 entries) so brute force is fine.
Distance metricCosine. Stored values are distance (0 = identical); we convert to similarity via 1 − distance.
What's indexedEach fragment's name + category + doc string is embedded once when the registry is synced. Sync runs lazily before each search and skips fragments whose content hash hasn't changed.

The index is rebuilt only when a fragment's doc or name changes. The fragment library is versioned by content hash; an edit triggers a re-embed for that one entry on the next search call.

4 Scoring logic — line by line

Every match carries two numbers, both rounded to two decimal places:

// fragment-index.ts:344-355
const best = 1 - rows[0].distance;            // top match's raw similarity
const matches = rows.map((row) => ({
  similarity: round(1 - row.distance),         // absolute cosine sim, 0..1
  score:      round(best > 0 ? sim / best : 0), // relative — winner = 1.0
  ...
}));
NumberWhat it meansRange
similarityRaw cosine. "How close is this fragment in vector space, ignoring others."0..1
scoreSimilarity divided by the best match for THIS query. "How good is this relative to the winner."0..1 (winner = 1.0)

Two-threshold filter

// fragment-index.ts:357-361
const passing = matches.filter(
  (m) => m.score >= minScore                      // ≥ 0.8 (default)
      && m.similarity >= ABSOLUTE_SIMILARITY_FLOOR // ≥ 0.35
);
if (passing.length === 0) {
  // Nothing qualifies → return ONLY the best, flagged.
  return [{ ...matches[0], belowThreshold: true }];
}
ThresholdDefaultWhat it catches
Relative score ≥ 0.80.8 Long-tail mediocre matches. If best = 0.55 and a sibling is 0.42 (≈76% of best), it's filtered. Only the top tier survives.
Absolute similarity ≥ 0.350.35 "The whole library is off-topic." If best = 0.25, every relative score is ≥0.8 (they share the same low ceiling) — but the absolute floor catches it. The result is returned belowThreshold: true.
Why 0.35 absolute? Measured empirically with text-embedding-3-small: on-topic hits cluster around 0.45-0.6, unrelated queries around 0.2-0.27. 0.35 splits the distributions cleanly.

5 Gap classification

A section is a gap when the library has nothing usable. The analyser is what tells the developer where to invest fragment-authoring effort next.

// fragment-coverage.ts:102-112
const top = matches[0];
const gap = !top || top.belowThreshold === true;
sections.push({
  page: page.name,
  section,
  fragmentId:   gap ? null : top.id,
  fragmentName: gap ? null : top.name,
  score:        top ? round(top.similarity) : 0,  // raw cosine, not relative
  gap,
});
ConditionResult
No matches at all (empty library)gap
Top match below ABSOLUTE_SIMILARITY_FLOOR (0.35)gap
Top match passes both thresholdsnot a gap; fragmentId set

The score shown in the analyser table is the raw cosine similarity, not the relative one. So 0.52 reads as "52% cosine — solid on-topic hit"; anything in the gap list had its best candidate below 0.35.

7 Outputs — two consumers

UI consumer components/builder/fragment-analyzer.tsx

  • Top-line — "{coveragePct}% of sections have a close fragment · {n} fragments suggested to the builder"
  • Gap callout (when any gaps) — amber-tinted list of {page} › {section} rows: "built from primitives, and candidates for new fragments"
  • Per-section table — three columns: section · page, matched fragment id (or "— build from primitives"), and the raw similarity score (or "gap").

8 Rerun semantics

The cache key intentionally ignores the fragment library — adding a new fragment doesn't invalidate it. The Rerun button is the manual override.

ActionWhat changesCost
Open analyser (GET)Serves cached coverage if sitemap-hash matches; otherwise computes once.Free on cache hit; N embedding calls on miss (N = total sections)
Edit a section/purposeSitemap hash changes → next view auto-recomputes.N embedding calls
Rerun button (POST)clearFragmentCoverage(appId) wipes the cache file, then recomputes.N embedding calls + a duration chip surfaces the wall-clock cost

Use the Rerun button when you add or edit fragments and want to see updated coverage without changing the sitemap.

9 Tuning levers (for the next iteration)

Every knob that could meaningfully change the result, with the trade-off it controls. Most are one-line edits — saved here as the catalog of moves we can make.

LeverWhereCurrentTrade-off when raised / lowered

Embedding modelfragment-index.ts:23openai/text-embedding-3-small (1536d) Larger models (e.g. 3-large at 3072d) give better separation but slower + costlier. The 0.35 floor would need re-tuning.

Open questions

  • Should the agent's per-page searchFragments calls share the same thresholds, or be stricter (since they have richer queries)?
  • Worth tracking per-section computeDurationMs so we can see which sections embed slow (network spikes)?
  • Should gaps over multiple apps be aggregated into a fragment-authoring backlog?
  • Try a hybrid score: (similarity + relativeScore) / 2 to reduce binary cliff at the floor?
