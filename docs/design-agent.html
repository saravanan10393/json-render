Design agent · v1 · how it works now

patchwork* — the Design agent

The middle agent in the staged pipeline. Owns the theme, the sitemap, and the per-page layout mockup the Frontend agent rebuilds from. Deliberately fragment-blind — designs at the level of intent. This documents the design agent as built. Sibling docs: understanding_v1.html (whole system), embedding-analyser.html (analyzer).

0 The design agent in one picture

It runs after the Backend agent's data model is approved and before the Frontend agent builds pages. Three tool calls (one per artifact) produce the three things the Frontend agent reads from the system context.

  • Position: backend → design → frontend. Stage advances on explicit human approval; it never runs as a side-effect.
  • Fragment-blind on purpose: no searchFragments tool. The agent designs at the level of UX intent — sections, copy, layout, state coverage — and the Frontend agent picks fragments later. This keeps the designer from over-fitting to whatever happens to be prebuilt.
  • One agent invocation per tool when run as a system action (rerun-theme, rerun-sitemap, generate-mockup-per-page). One bigger agent.stream call when triggered via chat. Both paths land in the same three artifacts.
  • Default model: google/gemini-3.5-flash (multimodal, 1M ctx, fast at text). Override per-app in the Models popover. Required: tool/function calling support.

1 Tools mastra/tools.ts

Three tools. Everything else (no defineEntity, no seedRecords, no savePage, no searchFragments) is deliberately absent — the design agent's blast radius is exactly these three artifacts.

ToolWhat it writesNotes
applyDesignSystem({ preset } | { colors, headingFont, bodyFont, radius }) theme.json (and DESIGN.md when authored from scratch) Two modes. PICK a preset from theme-presets.ts (32 entries, complete looks) — no DESIGN.md. CREATE authors a full token set; writes DESIGN.md, lints with @google/design.md, parses to theme.json.
saveSitemap({ pages, navigation, home, shellLayout, flows }) design/sitemap.json The information architecture. Every page gets an id, name, purpose, primary entity, ordered sections, and state coverage (empty / loading / error). Navigation shell is one of 6 layouts.
saveDesignArtifact({ pageId, mode, content }) design/mockups.json (per page · per mode) Mode is text (markdown layout) or html (static styled page). pageId must match a saveSitemap page id — orphan ids reject. Image mockups are generated by an external image-gen model; the agent never passes mode:'image'.
Why no searchFragments? Two reasons. (1) If the agent saw the fragment library it would design toward what exists, biasing every app toward the same handful of layouts. (2) The Analyzer step measures coverage after the design lands — gaps are valuable signal for what new fragments to build, and you only get that signal if the design wasn't anchored to the existing set.

What the prompt actually tells the agent

  • Role: "You are the Designer in a staged app-building pipeline — you act as a product designer."
  • Scope: design at the level of INTENT (layout, hierarchy, copy). No knowledge of components or fragments.
  • No emoji rule: never any emoji — not decorative (👋), not status (✓), not category (🔥). Every icon goes by its kebab-case lucide name (lucide.dev/icons): inline in text/html mockups as [icon:droplet]; in nav as the icon field on each entry.
  • Mockup ONLY the page CONTENT area — never draw sidebar/topnav inside a mockup. The runtime renders the nav shell from app.json + shellLayout; drawing chrome inside a page mockup leads the Frontend agent to build duplicate navigation inside each page.

2 Theme — pick or create

A theme is colors + fonts + radius (light & dark palettes). The agent either picks one of 32 ready-made presets, or creates one from scratch as a DESIGN.md spec.

  • 32 presets in lib/jr/theme-presets.ts — neutral default + 6 domain presets (commerce-warm, ops-utility, editorial, studio-bold, finance-trust, wellness-soft) + 25 tweakcn palettes. Each is a complete look: id · label · description · fonts · radius · light · dark.
  • Domain → preset guide baked into the prompt: shops/food/retail → commerce-warm · helpdesk/CRM/ops → ops-utility · blogs/content → editorial · project/creative → studio-bold · finance/legal → finance-trust · wellness/education → wellness-soft. The rest are brand/aesthetic palettes the agent picks by description.
  • CREATE mode contract: every REQUIRED_TOKEN (background, foreground, card, primary, …) must be set; prefix any with dark- for the dark palette. Fonts MUST come from the curated Google Fonts list (loaded by the runtime via <link>). Radius is a rem value in [0, 2].
  • DESIGN.md is only written on CREATE. PICK goes straight to theme.json (no spec round-trip). The tweaker's token/font/radius edits go through applyThemeEdit — also no DESIGN.md.

3 Sitemap — IA + navigation

The sitemap is a structural contract: every page the Frontend agent will build, in the order the navigation surfaces them, with explicit empty/loading/error coverage.

// sitemap.json shape (zod-validated by saveSitemap):
{
  "pages": [
    { "id": "dashboard", "name": "Today's Checklist",
      "purpose": "Track today's habit completions and view streaks.",
      "primaryEntity": "Habit",
      "sections": ["Greeting header", "Today's habits", "Streak summary"],
      "states": ["empty", "loading"],
      "icon": "layout-dashboard"      // lucide name
    },
    /* ... habits, journal, etc. */
  ],
  "home": "dashboard",             // id of the home page
  "shellLayout": "sidebar",        // 1 of 6 layouts (see table)
  "navigation": ["dashboard", "habits", "journal"],  // top-level only
  "flows": [
    "Add habit: dashboard → '+ New habit' → form → back to dashboard",
    "View streak detail: dashboard tile → detail drawer"
  ],
  "durationMs": 9450           // stamped by installDesignTiming
}

The 6 shell layouts

shellLayoutWhat it isUse it for
sidebarClassic left nav with labelsDefault for 3+ page business apps
topnavHorizontal header navConsumer-facing apps (shops, blogs)
icon-railSlim icon-only railDense ops/admin tools where space matters
compact-railIcons + tiny labelsDashboards with 4-6 sections
minimalNo chrome, floating page switcherSingle-purpose or landing-style apps
split-railIcon rail + secondary label panelLarge multi-module apps
  • Top-level navigation only: dashboards, lists, settings. Detail/form pages are reached via row clicks, not nav entries.
  • Every nav entry needs a lucide icon name — the runtime renders it via the Icon component (same library).
  • States matter: the empty/loading/error states a page declares are surfaced to the Frontend agent so it builds the right binding paths (e.g. an explicit empty-list panel, not a blank table).

4 Mockups — per page, three modes

Each sitemap page gets up to three mockup representations. They coexist — saving one (pageId, mode) keeps the others — and the human picks ONE mode as the "used for build" via the radio in the design review.

ModeWhat it isAuthored byBuild budget
text Concise markdown — sections top-to-bottom + state coverage, real copy Design agent · saveDesignArtifact 5,000 chars / page
html Full self-contained styled HTML — inline <style> using theme CSS vars, 1100px content width. Auto-renders to a PNG screenshot (see §6) cached alongside the html, attached to the build as a vision input so the agent gets both the markup AND the spatial truth. Design agent · saveDesignArtifact 40,000 chars / page + screenshot (~150-220 KB PNG attached)
image Generated PNG/JPEG (data:image/... URL) External — Gemini 3.1 Flash Image (see §5) Attached as image part (no truncation)
Why "content only"? The Frontend agent builds nav shells from app.json + shellLayout, not from the mockup. If the mockup includes a sidebar, the Frontend agent ends up duplicating navigation inside every page. The prompt forbids this explicitly: "Mockup ONLY the page CONTENT area — the body inside the nav shell."

Per-page parallel generation

generateMockupRepresentation() fans out across pages with Promise.all — one agent.generate per page (each maxSteps: 6), all running concurrently. Wall-clock is bounded by the slowest page, not the sum.

// builder-run.ts — generateMockupRepresentation
await Promise.all(
  targetIds.map(async (id) => {
    const page = sitemap.pages.find(p => p.id === id);
    const agent = makeDesignAgent({ model });
    const startMs = performance.now();
    await agent.generate(
      [{ role: "user", content: `Generate the ${mode} mockup for "${page.name}".` }],
      { maxSteps: 6, context: [{ role: "system", content: sysContent }], ... }
    );
    setMockupDuration(appId, page.id, mode,
      Math.round(performance.now() - startMs));   // stamp durationMs on the slot
  })
);

Mode-specific shaping

The per-page system prompt appends mode-specific guidance:

  • text: a markdown bullet list — sections in render order + a "states" subsection for empty/loading/error.
  • html: single <html> document, no <nav class="sidebar">, inline <style> referencing theme CSS vars (var(--background), var(--primary), …) so the preview matches the live theme. Body styled max-width: 1100px; margin: 0 auto; — the shell wraps it at runtime.

5 Image mockups — external lib/server/image-gen.ts

Image mode bypasses the design agent. A text→image model (google/gemini-3.1-flash-image by default) is called directly with a synthesized prompt built from the sitemap + theme.

  • Why external: text→image models don't expose tool calling — they can't call saveDesignArtifact. Image gen happens in a dedicated POST /v1/chat/completions with modalities: ["image"].
  • Default model: google/gemini-3.1-flash-image — overridable in the Models popover. The prompt is synthesized from the sitemap page (sections + purpose) and the theme (primary, background, fonts, radius) so the rendered image reflects the actual design system, not a generic mockup.
  • Storage: the response image is base64-encoded and stored as a data:image/... URL in mockups.json. No filesystem image file — keeps the snapshot self-contained.
  • Frontend handoff: when image is the selected mode, runStageTurn attaches the data URL as a type: "file" message part. Sonnet 4.5 (the default frontend model) reads it natively as vision input.

6 HTML screenshot pipeline lib/server/html-renderer.ts

Every html mockup the agent saves also gets rendered to a PNG and cached on disk. The rendered image is handed to the frontend agent alongside the inlined HTML text — multimodal: text for copy fidelity (verbatim quotes), image for spatial truth (proportions, alignment, visual rhythm). Neither alone is enough; together they close the design-to-code failure modes that pure-text or pure-image handoffs leak.

How it fires

The hook lives inside the saveDesignArtifact tool execute: right after writeMockup(..., "html", content) persists the html, a void cacheHtmlScreenshot(appId, pageId, content) kicks off the render asynchronously and returns immediately. The tool result goes back to the agent without waiting; the screenshot lands on disk a couple of seconds later. The next rebuild that runs with selected === "html" reads the cached file and attaches it.

// mastra/tools.ts — saveDesignArtifact.execute (excerpt)
writeMockup(appId, input.pageId, input.mode, input.content);
stampDesignTiming(context, ms => setMockupDuration(appId, input.pageId, input.mode, ms));
touchApp(appId);

// Async render — does not block the tool response.
if (input.mode === "html") {
  void cacheHtmlScreenshot(appId, input.pageId, input.content).catch((err) =>
    console.warn(`[saveDesignArtifact] html→png cache failed for ${input.pageId}:`, err)
  );
}

The renderer

renderHtmlToPng() uses puppeteer-core with the system Chrome binary — not the bundled Chromium. That's a ~165 MB lighter install at the cost of one assumption (Chrome present on Mac, where it almost always is). The CHROME_PATH env var overrides the auto-detected location for CI / non-standard setups. The browser instance is cached across renders inside a process so puppeteer's ~400 ms launch cost is amortized after the first page.

  • Viewport: 1400×900 desktop by default (matches what the user sees in design review). deviceScaleFactor: 1 keeps PNGs around 100–250 KB; bump to 2 if Retina sharpness matters more than file size.
  • Wait condition: setContent with waitUntil: "domcontentloaded" plus a 300 ms settle. The alternative (networkidle0) would hang on external font CDNs the mockup may reference.
  • Full-page: screenshots use fullPage: true so long mockups aren't viewport-clipped.
  • Graceful disable: when no Chrome binary is found, renderHtmlToPng returns null; the saved mockup still works, the build just falls back to text-only handoff for that page (better than failing the save).

Cache layout

// Per-app screenshot cache — sibling to the existing design/ artifacts.
data/<appId>/
  design/
    mockups.json
    mockup-screenshots/
      <pageId>.png    // auto-rendered after each html save · ~150-250 KB
      ...
Why a cached file, not inline base64. Three reasons. (1) Keeps mockups.json lean — 5 inlined screenshots would push the JSON past 1 MB. (2) Lets a future "Open mockup" link in the UI serve the PNG directly. (3) Easy A/B: re-running the renderer with a different viewport / scale just overwrites the file; no schema work.

7 Parallel orchestration runDesignParallel

When the user clicks "Rerun all design" (or design runs as a system action), the three artifacts are produced in two parallel waves — theme + sitemap concurrently, then all per-page mockups concurrently.

  • No data dependency between theme and sitemap → fire them as two concurrent agent.generate calls. Each is a single maxSteps: 8 turn.
  • Mockups wait on the sitemap only (they need page ids). Once the sitemap lands, every page's mockup is generated concurrently via the same parallel-per-page mechanism as standalone mockup regen.
  • Why bother? Sequential "all" takes theme + sitemap + max(per-page mockup). Parallel takes max(theme, sitemap) + max(per-page mockup) — measured ~28% wall-clock savings on a 5-page app.
  • Returned timings: { theme, sitemap, mockups, total } in ms. The builder UI shows the breakdown as a banner after the rerun finishes.

8 Timing capture installDesignTiming

Every artifact carries a durationMs stamp — useful for spotting slow stages without opening logs. Two paths: chat-driven (tool-level instrumentation) and system-action (wall-clock outside the agent).

PathHow duration is measuredWhere stamped
Chat turn installDesignTiming(requestContext) records a baseline at agent start; each tool's execute hook computes now - lastTimestamp at completion (so theme = think+output time, sitemap = time between theme done and sitemap done). Tool's writer (applyDesignSystem → theme.json, saveSitemap → sitemap.json)
System action (rerun-theme / rerun-sitemap) Wall-clock wrapped around the entire agent.generate call. Same writer, same shape — UI doesn't care which path produced it.
Per-page mockup Wall-clock wrapped around each agent.generate in the parallel fan-out. setMockupDuration(appId, pageId, mode, ms) patches the slot.

In the UI: a small monospace chip next to the artifact title — Theme shows 3.2s, Sitemap shows 9.5s, each mockup tab shows its own (e.g. html 17.4s).

9 Multimodal handoff to the Frontend agent

Once the human approves, the Frontend agent reads three things from its system context: the data model (entities + sample rows), the theme summary (preset/fonts/radius), and the approved sitemap + selected mockup.

// builder-run.ts — formatMockups (the block the Frontend agent receives):

APPROVED DESIGN MOCKUPS (html, one per page) — build each faithfully in real components:

### Page: dashboard
<!DOCTYPE html>
<html> ... 18,000-40,000 chars of inlined html per page ... </html>

### Page: habits
...
  • Text mode handoff: the markdown is inlined verbatim into the system prompt (5KB/page budget). Distinct copy shows up byte-identical in the built spec — the Frontend agent quotes it directly. Single-modal: no vision attached.
  • HTML mode handoff (multimodal): the html source is inlined into the system prompt (40KB/page) AND the cached PNG screenshot (§6) is attached as a type: "file" vision part on a separate user message. Multimodal beats either alone — text alone misses spatial layout, image alone OCRs copy badly. The pair gives the Frontend agent verbatim copy fidelity AND faithful layout reproduction.
  • Image mode handoff: no text inlined; the generated PNG is attached as a vision part. Sonnet 4.5 (the default frontend model) reads file parts natively — no model swap.
  • The sitemap also flows through as JSON so the Frontend agent knows the page ids, sections, states, and the navigation shell to build.
  • Single helper, three modes: buildImageMockupMessage(appId, sel) in lib/server/builder-run.ts handles all three handoff shapes — image (data URL), html (read cached PNG from disk + base64 encode), text (returns null → text-only). Shared by both the chat path (runStageTurn) and the system-action rebuild path (runRebuild).
  • Built page ids come from the page NAME, not the sitemap id: savePage derives the file id as <role>-<entity>-<slug(name)>. So sitemap dashboard with name "Today's Checklist & Coach" ends up as file user-checkin-todays-checklist-coach.json. The mapping is deterministic but not visually obvious.
Measured impact of the multimodal handoff. On a 3-page habit-tracker test (same Sonnet 4.5 model, same design artifacts, only the selected mode changes):
  • text-mode build reproduced ~55% of the intent's section count, ~30% of titles verbatim, swapped card+form patterns for tables, missed entire sections.
  • html-mode build (text + screenshot) reproduced ~100% of section count, ~100% of titles verbatim, matched the intent's pattern choices (cards + filter pills + inline create form), and flagged the one component it genuinely couldn't build (heatmap grid) as an explicit placeholder in-spec.
