The agentic design agent

A target architecture for an AI design agent inside an application-building platform — what it should be composed of, what artifacts it owns, what loops it runs, where humans gate it, and how to evaluate its output. Tool-agnostic. Prescriptive, not a survey. Scope: only the parts an autonomous agent can do well today; classic designer responsibilities that should stay human are explicitly called out in §9.

0 The north star

A design agent's job inside an app-building platform is to convert intent (a brief, a domain, a few sentences) into durable, machine-readable design artifacts that a downstream build agent can faithfully implement, while ceding the parts that need real users or human taste back to a human at well-defined gates. The agent designs at the level of intent — sections, hierarchy, copy, interaction — not at the level of components or framework code.

  • The handoff is the product. Everything else — the model choice, the prompts, the agent topology — exists to produce three durable artifacts a build agent can read deterministically. Optimize for that handoff.
  • Design at the level of intent. The agent emits sections, copy, hierarchy, state coverage, navigation. It does not emit components, framework code, or implementation details — those are the build agent's job. This split keeps the design agent honest and the build agent free to choose components.
  • Grounding beats freedom. An agent grounded in an existing design system reuses components and produces higher-quality code; ungrounded, it wastes effort searching and tends to invent duplicates. (Figma MCP)

1 The phases — what a designer does, agent-restricted

A strong product designer's workflow has eight phases. An AI design agent can own four of them well, can partially own two, and should stay out of two altogether. Below: each phase mapped to the agent slice, what's produced, and what gates it.

PhaseWhat the designer doesAgent ownershipArtifact produced
1. Brief intakeCapture domain, audience, brand, constraints. Partial — agent extracts what it can from the brief; ambiguous brand/taste choices stay human. brief.md + structured intent
2. ResearchUser interviews, surveys, competitive analysis. Human-only. Real users, real interviews. (out of scope for the agent)
3. IAWhat pages exist, how they nest, how users navigate. Agent. The agent excels at decomposing a domain into pages, sections, and a nav shell. Sitemap (pages · sections · navigation · flows)
4. Visual languagePick colors, fonts, radius, density. Pick or author a theme. Agent, with human taste gate. Picks from a preset library; can author from a tokens spec. Design tokens (colors · fonts · radius · spacing · semantic states)
5. Wireframe / mockSketch per-page sections, real copy, layout intent. Agent. Produces multimodal per-page mockups (structured + visual). Per-page mockups (text · html · image)
6. CritiqueSelf-review and peer review against heuristics. Agent, as an LLM-as-judge loop grounded in named guideline sets. Critique findings + revision proposals
7. Usability testingWatch real users complete tasks; iterate. Human-only. Real users, real tasks. Synthetic critique is not a substitute. (out of scope)
8. Dev handoffAnnotate components, states, spacing for engineers. Agent. Produces structured artifacts a build agent reads. Component / section specs + handoff bundle
The four phases the agent owns well — IA, visual language, wireframing, dev handoff — share one property: the output is structured, machine-writable, and verifiable against the inputs. Where the output requires taste signals (brand voice) or real-user signals (usability), defer.

2 The three durable artifacts

A design agent is defined by what it writes. Three artifacts — and their schemas — are the spine of the whole system. Choose them carefully; everything else is plumbing.

2.1 · Design tokens — the visual language

The W3C Design Tokens Community Group format (DTCG) is the de-facto interoperability standard. Tokens are stored as JSON; the minimal token is a JSON object containing a reserved $value property, with optional $type and $description. The spec's explicit goal is to eliminate bespoke glue between design and build tooling — which is exactly the agent-to-build-agent problem. (designtokens.org spec)

// tokens.json — DTCG shape (any object with $value IS a token)
{
  "color": {
    "brand": {
      "primary":    { "$value": "#6366f1", "$type": "color" },
      "on-primary": { "$value": "#ffffff", "$type": "color" }
    }
  },
  "size": {
    "radius": { "md": { "$value": "0.5rem" } },
    "font":   { "body": { "$value": "1rem", "$description": "default body text" } }
  }
}

Tool ecosystems like Style Dictionary already compile a single token source into platform-specific style definitions (CSS/SCSS/iOS/Android), so picking DTCG as the output format means the agent's tokens are immediately consumable across stacks. (Style Dictionary)

2.2 · Sitemap — the information architecture

A structural contract: every page that will exist, in the order the navigation surfaces them, with the sections and intent baked in. The build agent reads this to know what pages to build.

// sitemap.json — structural, zod-validatable
{
  "pages": [
    {
      "id": "dashboard",
      "name": "Today",
      "purpose": "At-a-glance: what's due today and recent activity.",
      "primaryEntity": "Task",
      "sections": ["Header with greeting", "Today's tasks", "Recent activity"],
      "icon": "layout-dashboard"
    }
  ],
  "home": "dashboard",
  "shellLayout": "sidebar",
  "navigation": ["dashboard", "tasks", "settings"],
  "flows": ["Create task: header → '+ New' → form → back to dashboard"]
}

2.3 · Per-page mockups — multimodal

Per-page wireframes / mockups, one per sitemap page, in the representations the build agent can actually parse. The mockup is content only, never the nav chrome — the shell is rendered by the runtime from the sitemap. Trade-offs between text, structured, and image representations are detailed in §5.

Why these three. Tokens own the visual language, the sitemap owns the structure, and the mockups own per-page intent. Together they're the minimal complete contract: visual + structural + content. Everything else a designer makes (annotations, redlines, specs) is derivable from these three.

3 Agent decomposition — single, planner+executor, or multi-agent

Three topologies are viable. Pick by task parallelism, not by ambition.

The recommended baseline: plan + execute

A planner LLM produces a multi-step plan up front; an executor LLM runs each step with the agent's tools; a re-planner decides after execution whether to finish or generate a follow-up plan. This is a stronger default than reactive (ReAct) because (1) the planner reasons about the whole task instead of one sub-problem at a time, and (2) it doesn't need a new LLM call per tool invocation. (LangChain · planning agents)

When to escalate to multi-agent

Multi-agent decomposition (orchestrator + parallel specialist subagents) is a real win — Anthropic measured a 90.2% improvement over a single-agent baseline on research-style tasks (Anthropic multi-agent) — but only when (a) the work is genuinely parallelizable and (b) the subagents don't need to share each other's context. Most design phases share context (a visual choice in §4 constrains the mockups in §5), so the natural multi-agent split is per-page mockup generation, not cross-phase.

Subagent contract. When you do spawn subagents, each one must be given a concrete contract: objective, output format, tool/source guidance, and explicit task boundaries. Skipping any of these is the dominant cause of subagent failure and duplicated work. (Anthropic multi-agent)

Practical recipe

  • For 1-3 pages: single agent with the three tools. Latency wins.
  • For 4+ pages: plan + execute, with per-page mockup generation fanned out in parallel. (Mockups are embarrassingly parallel — each page only needs the sitemap + tokens, not other pages' mockups.)
  • For larger work (deep IA, design audit, redesign): orchestrator + specialist subagents (IA, visual, per-page mockup, critic). Use the subagent contract above.

4 The tool surface — minimum viable

A small, sharp tool set. Each tool writes exactly one of the three artifacts (or a slice of one), so the agent's blast radius is always knowable.

ToolWritesInput shapeNotes
applyTokens tokens.json Either a preset id (pick) or a full { colors, fonts, radius, ... } token set (author) Two-mode tool. Pick is preferred — leans on a curated library. Author is the escape hatch when no preset fits.
saveSitemap sitemap.json Pages array + nav + home + shellLayout + flows Schema-validated. Rejects orphan refs (e.g. a flow referencing a non-existent page).
saveMockup mockups/<pageId>.{text,html,image} { pageId, mode, content } — pageId must match a sitemap page One mode at a time; multiple modes coexist. See §5 for representations.

Tools the design agent should NOT have

  • No searchComponents / fragment library access. If the agent sees the existing component library, it designs toward what exists — every app drifts toward the same handful of layouts. The agent designs at the level of intent; the build agent picks components. (This isn't ideological — it's an explicit information-architecture choice that produces more diverse designs and surfaces gaps in the component library as actionable signal.)
  • No savePage / codegen tools. The design agent does not write framework code. If you can't cleanly separate this from the build agent's surface, your boundary is wrong.
  • No write access to data / business logic. The data model lives upstream; the design agent reads it.
The fewer tools the better. Three tools, three artifacts. Resist the temptation to add tools for “a designer would do that” — most of those tasks reduce to the existing artifacts. Adding tools expands the agent's blast radius without expanding its capability.

5 Mockup representation — the central trade-off

How do you represent a mockup so a downstream build agent can reproduce it? The choice between text, structured JSON, and image dominates handoff quality. The honest answer is multimodal: structured + image together.

RepresentationCarriesStrengthsBlind spots
Text (markdown) Section order, real copy, state coverage Cheap to generate, easy to diff/edit, build agent can quote copy verbatim No spatial signal — relative widths, alignment, density are invisible
Structured JSON
(condensed scene graph)		 Semantic (text · element type · group name) + visual (x, y, w, h, color, font) per node Parseable, queryable, fits in context (~3-5k tokens per UI); precise positions Cannot represent “does this element actually overlap that one?” or center-alignment of differently-sized elements without rendering it. (Duan et al. 2024)
Image (screenshot) The actual rendered visual — color, density, balance, rhythm Captures everything the structured representation misses; a vision-capable build agent reads it natively Not queryable; text is OCR-lossy; large in tokens; can't reliably extract copy verbatim

The multimodal recommendation

Combining a screenshot with structured metadata produces better code than either alone — this is the central empirical finding from the design-to-code literature. (Figma MCP) A design agent should produce both a structured representation (for queryability + copy fidelity) and a rendered image (for spatial fidelity), then hand both to the build agent.

The dominant failure mode in design-to-code

Empirical work on design-to-code benchmarks finds the dominant failure isn't styling — it's recalling which visual elements exist in the input and reproducing correct layout. The implication for a design agent: the handoff artifact should explicitly enumerate elements (the structured representation's job) rather than trust the build agent to infer them from an image alone. (Si et al. Design2Code)

What goes in the structured representation

Each node should carry both semantic and visual data. The condensed JSON scene-graph approach from Duan et al. is the canonical shape: { type, name, text, x, y, w, h, color, bg, opacity, font, children }. Trimmed of decorative noise, a complete page fits in ~3-5k tokens. (Duan et al. 2024)

Practical posture. Default to structured JSON + image as a pair. Drop the image (cost) only when the design is genuinely spatial-trivial — a one-section form, a settings list. Drop the structured (no copy fidelity) only when the agent literally cannot produce machine-readable output.

6 The three loops — plan, critique, validate

A design agent is not one-shot. It runs three nested loops at different granularities.

6.1 · Outer loop — plan / execute / re-plan

The plan-and-execute pattern from §3. The outer loop owns the decision “am I done, or do I need another plan?” — and is the natural place to attach token budgets and step caps.

6.2 · Middle loop — self-critique (reflection)

For each artifact draft, run a critique pass before saving. The critic LLM (same agent in self-critique, or a separate “critic” subagent) reviews the draft against a fixed rubric and emits revision proposals; the drafter incorporates them. Two-pass minimum: draft → critique → revise.

The critical insight from the literature: grounding the critic in specific, named guideline sets — Nielsen's 10 Heuristics, accessibility checklists, named visual design principles — yields higher-quality and more diverse feedback than asking for “general UI feedback”. (Duan et al. 2024)

6.3 · Inner loop — validators (deterministic)

Before the agent ever sees a draft as “done,” deterministic validators reject malformed output:

  • Schema validation on tokens.json (DTCG conformance), sitemap.json (zod), mockup metadata.
  • Referential integrity: every flow references real pages; every navigation entry exists; every primaryEntity exists in the data model; every icon name is in the icon library.
  • Accessibility primitives: contrast ratios on token pairs (WCAG AA/AAA), font-size minimums, target-size minimums.
  • Cross-artifact consistency: every mockup's pageId matches the sitemap; every section referenced in a mockup is declared in the sitemap.
Order matters. Validators run first (cheap, deterministic). If they fail, the agent retries that artifact without burning critique tokens. Critique runs only against validator-passing drafts.

7 Design QA — the critic's rubric

The middle loop's critic is only as good as its rubric. Five dimensions, each grounded in a named guideline set rather than a vague “is this good UI?” prompt.

DimensionGrounded inWhat the critic checks
Usability Nielsen's 10 Usability Heuristics Visibility of system status · match real-world language · user control · consistency · error prevention · recognition over recall · flexibility · aesthetic-minimalist · error recovery · help.
Visual hierarchy Named visual design principles (e.g. CrowdCrit) Is the primary action visually dominant? Are sections clearly separated? Does the eye follow the intended path?
Semantic grouping Named semantic-grouping guidelines (e.g. Duan et al.) Are related elements visually grouped? Do labels sit near their controls? Are sections coherent?
State coverage Loading / empty / error / partial state checklists Does every list have an empty state? Every async surface a loading state? Every form an error path?
Accessibility WCAG 2.1 AA Color contrast on token pairs · target sizes · semantic structure · keyboard reachability hints in mockup.

How the critique step is structured

// Critique prompt — grounded in a NAMED rubric (not "general feedback")
You are a UX critic. Review this mockup against Nielsen's 10 Usability Heuristics
ONLY. For each heuristic that applies, return:
  - heuristic: <name>
  - violation: <concrete description>
  - severity: critical | major | minor
  - revision: <specific change to make>

If a heuristic doesn't apply to this artifact, omit it.
Why named rubrics matter. The Duan et al. result is concrete: specific guidance produces higher-quality and more diverse critique than open-ended “tell me what's wrong.” Pick a rubric, name it, and instruct against it. If you want multi-dimensional critique, run the critic multiple times with different rubrics, not one omnibus pass.

8 Approval gates — where the human signs off

An agent that ships without human review is a tool, not a colleague. Three explicit gates, between phases. Each gate is small (review one artifact, approve / request changes); the agent never advances past a gate on its own.

The three gates

  • Gate 1 — Brief approval: did the agent capture intent correctly? Catches misunderstandings before they compound through three artifacts.
  • Gate 2 — Structure approval: tokens + sitemap. Visual language and IA are the architectural decisions; wait to spend mockup tokens until the human has signed off.
  • Gate 3 — Design approval: per-page mockups. Last point before handoff to the build agent.

Gate UX

Each gate has three actions: approve (advance), request changes (specify what), reset (start the phase over). The agent does not advance the run state without an explicit approve. Critique can be auto-run by the agent before presenting at the gate, but the gate itself is human.

Why gates, not just review-and-trust. Even with strong critique loops, the literature on automated UI design quality is honest: synthetic critique cannot substitute for real human taste on brand/voice and cannot substitute for real users on usability. Gates exist to keep humans authoring the parts they should still author.

9 What stays human — the explicit out-of-scope

The scope discipline that makes this whole architecture honest. Five responsibilities a strong designer owns that an autonomous AI design agent should not attempt to own.

ResponsibilityWhy it stays humanWhere the agent helps anyway
User research Requires real users, real interviews, real ethnographic observation. Synthetic personas drift from real ones. Agent can summarize known research artifacts as context; can flag where data is missing.
Product strategy What to build, for whom, against which alternatives. Requires market judgement and stakeholder alignment. Agent can produce options matrices and consequence analyses; the call stays with humans.
Brand & taste sign-off Brand voice and aesthetic judgement are taste calls — culturally and temporally specific. Agent picks from a brand-curated preset library and proposes; the human approves at Gate 2.
Usability testing Real users completing real tasks. Synthetic LLM-as-judge is not a substitute for observation. Agent runs pre-flight heuristic checks (§7); humans run the actual sessions.
Final accessibility audit Automated checks catch ~30-40% of WCAG issues; the rest needs trained human auditing and assistive-tech testing. Agent enforces token-level contrast, target sizes, and semantic structure as validators; humans audit before ship.
The point. A design agent that respects these five boundaries is a sharp colleague. One that doesn't is a liability. The boundary is not about capability — it's about what humans should own because the consequences are theirs.

10 Evaluation — how you know the agent is good

An agent without an eval loop is a guess. Four evaluation methods, ordered from cheap-to-run to high-signal. Use them in combination.

10.1 · Schema + invariant checks (cheap, deterministic)

Every artifact has a schema. Every cross-artifact relationship has a referential-integrity check (every flow points at a real page, every mockup matches a sitemap pageId, every token is DTCG-valid). These are pass/fail and run in milliseconds. The cheapest possible signal: is the output structurally well-formed?

10.2 · LLM-as-judge against a named rubric (medium cost, high diversity)

The critique-loop rubrics from §7, run as standalone evals. Score each artifact 1-5 per heuristic; aggregate across runs. Two practical notes: (a) judges grounded in named rubrics (Nielsen, WCAG) are far more reliable than open-ended “is this good?”; (b) self-judge bias is real — use a different model for the judge than for the drafter.

10.3 · Visual diff & design-to-code fidelity (high cost, high specificity)

Render the agent's mockup, render the build agent's implementation, diff them. The design-to-code literature decomposes this into specific sub-metrics: high-level visual similarity (CLIP image embedding distance) + low-level element matching (block-match, text-OCR match, position match, color match). This is the metric suite a faithful handoff should optimize. (Si et al. Design2Code)

10.4 · Real users (highest cost, ground truth)

Methods 10.1–10.3 are proxies. Real-user testing is the ground truth and stays human-owned (§9). Use the cheaper methods to filter aggressively before spending real-user time.

The eval triangle. Schema invariants catch broken output. LLM-as-judge catches obvious craft issues. Visual diff catches handoff fidelity. Real users catch the rest. Each layer assumes the layer below has already filtered out its class of failure; without this stratification, evals are noisy and slow.
