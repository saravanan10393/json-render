patchwork* — how the builder works now

The staged design pipeline is live. One agent that did everything is now an orchestrated pipeline of three agents — Backend · Design · Frontend — with human approval gates between them, a view-only Analyzer step, and a real product-designer stage that owns the theme, the sitemap, and a per-page layout mockup the frontend rebuilds from. This documents the system as built. Sibling docs: understanding.html (before) and design-pipeline.html (the plan).

0 The pipeline in one picture

One conversation, routed. The orchestrator reads the run's stage and hands each chat turn to the right agent. Two human gates bracket the Design stage; a view-only Analyzer sits between Design and Build. Each stage persists its own artifacts; the Frontend agent rebuilds the real app from the data model + the approved design.

  • The whole pipeline is two toggles wide: Designer on/off and Fragments on/off. Designer off = the original Backend → Frontend path; the Design & Analyzer steps disappear and the app loads the neutral default preset theme — no LLM picks a theme, but it's still fully editable by hand in the theme tweaker.
  • Nothing here replaces the runtime. The Frontend agent still writes the same json-render page specs the react-router runtime renders against real local datasources — the pre-pipeline contract is unchanged.
  • The combined appBuilderAgent still exists, unused by the UI, kept as the benchmark baseline.

1 Run state machine lib/server/runs.ts

One runs row per app holds the current stage and the config toggles. Stages double as their own approval gates, which keeps the orchestrator a lookup rather than a workflow engine.

  • STAGES = backend → design → frontend → done. nextStage() honours the designer toggle; isGate() marks backend and design as the two stages that wait for a human Approve.
  • RunConfig = { designer, fragments }. ensureRun() infers a sensible starting stage for pre-pipeline apps (pages already on disk → frontend).
  • Controlled by /api/apps/[id]/run: approve (advance), reset (back to backend), reopen {stage} (jump the live stage earlier to re-run it), and config (toggles).

2 Orchestrator = the chat route /api/apps/[id]/chat

There is no separate orchestrator agent (decision D4: keep it simple). The chat route is the orchestrator: it reads the run, picks the stage's agent, builds stage-scoped context, and streams the turn.

  • Every chat turn calls runStageTurn({ appId, messages, stage, config }) in lib/server/builder-run.ts.
  • agentForStage(stage) selects Backend / Design / Frontend; buildBackendContext / buildDesignContext / buildAppContext assemble only what that stage needs (entities, theme, prior design, sitemap).
  • The user always talks to the orchestrator — there is no direct user→agent line. Each agent sees a curated context, not the raw history of the others.

Why fragment-blind design? Fragments exist to cut build time and lift quality — they are a frontend concern. If the designer knew them, it would design toward them. Instead the design is made freely, then the Analyzer (§6) measures how close it lands to what we have.

4 What the Design agent produces

Three durable artifacts, each a contract for a later stage. The mockup is a "rebuild-from-reference" — a picture/markup the frontend matches, not code it copies.

  • Theme — usually just theme.json picked from a preset; DESIGN.md is only written when the agent creates a theme from scratch (one path, see §5). Applied as scoped CSS vars at runtime.
  • Sitemap — the structural contract: every page (id, name, purpose, sections, states), navigation, home, shellLayout. The frontend agent builds against it; the Analyzer reads it.
  • Mockups are per-page — mockups.json = { selected, pages: { [pageId]: { text?, html?, image? } } }. Each slot also carries durationMs (stamped at write time — see §9). The human picks which mode is "used for build". Generation is a system action (§9), fanned out per page in parallel. text and html are written by the design agent's saveDesignArtifact tool; image bypasses the agent entirely — a direct text→image call to google/gemini-3.1-flash-image with a prompt synthesized from the sitemap + theme.
  • Per-tool timing — each design tool stamps durationMs onto its artifact at fire time (theme, sitemap, mockup). Works for chat-driven turns AND the scoped Rerun buttons. See lib/server/design-timing.ts.

5 The theme system

One preset library, three modes. A preset is a complete look (colors + fonts + radius, light+dark) that both the design agent and the human pick by description. The agent either picks a preset or creates one from scratch; the human picks or edits live in the tweaker. DESIGN.md is written on one path only — create-from-scratch.

  • One library — 32 presets in lib/jr/theme-presets.ts: the neutral default baseline, the 6 domain presets (commerce-warm · ops-utility · editorial · studio-bold · finance-trust · wellness-soft, folded in from the old design/presets/*.DESIGN.md — they kept their curated fonts), and 25 real tweakcn palettes (cyberpunk, claude, vercel, supabase, …). Every entry is a complete look: id · label · description · fonts · radius · light · dark. themePresetReference() (in theme-catalog.ts) feeds the agent's pick list and the tweaker tile tooltip from the same descriptions.
  • PICK — applyDesignSystem({ preset }) → pickThemePreset builds theme.json straight from the entry. No DESIGN.md. (The tweaker's tile click hits this path too, replacing the theme's identity.)
  • EDIT — the human tweaker's token/font/radius changes → applyThemeEdit overlays them on the current theme in place. No DESIGN.md; identity preserved.
  • CREATE from scratch — only when no preset fits: applyDesignSystem({ colors, headingFont, bodyFont, radius }) → authorThemeFromScratch writes a DESIGN.md spec artifact (per the design.md schema), lints it, then parses to theme.json. The one and only DESIGN.md writer.
  • The tweaker (theme-tweaker.tsx) — preset grid (active-state highlighted), deterministic Shuffle, grouped token editing, font dropdowns, a radius slider, instant preview, all via /api/apps/[id]/theme.
  • The same token set + font list + radius range is documented to the agent (mastra/instructions.ts) and shown in the tweaker — one shared source of truth in lib/jr/theme-options.ts.
  • Chart tokens — every preset defines chart-1..5. For the 9 presets that don't (default, the 6 domain presets, neumorphism, claymorphism), themeVars() (lib/runtime/app-shell.tsx) derives a ramp from primary at render time (small ±hue + lightness shifts) so charts inherit the theme's hue family instead of falling back to grayscale. Recharts cells read the var via inline fill="var(--chart-1)".
  • Shadows + categories — presets are grouped by visual category (Minimal · Soft · Bold · Editorial · Brutalism · Neumorphism · Claymorphism). Each category ships its own shadow-2xs … shadow-2xl scale wired to the Tailwind shadow-* utilities (flat shadcn, soft blur, hard offset, dual-shadow, puffy inset). The tweaker preset grid is grouped by category for browsing.

6 The Analyzer /api/apps/[id]/analyzer

The bridge between a fragment-blind design and a fragment-aware build. A view-only step (not a run stage) that measures how well the approved design maps onto our prebuilt fragments — and surfaces gaps.

  • computeFragmentCoverage() (lib/server/fragment-coverage.ts) does a semantic match of each sitemap section against the fragment library, cached by sitemap hash to design/coverage.json.
  • The result is preponed into the Frontend agent's context as a head start ("these sections likely map to these fragments"); a 0% match is a signal to the developer that we are missing a block.
  • Rendered in the builder as the read-only Analyzer step (fragment-analyzer.tsx), between Design and Build.
  • Rerun button — same sitemap = always serves cached results (no embedding cost). The Rerun button in the analyzer header POSTs to /api/apps/[id]/analyzer, which wipes coverage.json, recomputes from scratch, and returns the elapsed wall-clock. Used when the fragment library changes — sitemap-driven invalidation can't see that.

7 The Frontend handoff — building from the picture

On Build (or Rebuild), the Frontend agent receives the requirements, the data model, the sitemap, the fragment coverage, and the approved mockup. The handoff shape depends on which mockup mode the human picked:

Mockup modeWhat's in the agent's promptHow the agent reads it
textInlined markdown per page in the system prompt (5 KB/page budget)Plain text — sections + states + real copy
htmlInlined HTML per page in the system prompt (20 KB/page budget)Static HTML/CSS, content area only (max-width 1100px)
imageSystem prompt announces "vision input on next message"; a synthetic user message carries one {type:"file", mediaType:"image/...", url:<data URL>} part per pageSonnet 4.5 (default frontend) is vision-native — reads each PNG/JPEG directly. No model swap, no rasterisation, no sharp.
  • Image generation flow — when the design step's selected mode is image, the design agent is bypassed for the per-page work. generateMockupRepresentation() synthesises a single-shot prompt (page name + purpose + sections + states + theme name/primary/fonts/radius) and calls generateMockupImage() (lib/server/image-gen.ts) — a direct /v1/chat/completions POST with modalities: ["image"] to the configured image-gen model. Default: google/gemini-3.1-flash-image. The returned base64 PNG is stored as a data:image/...;base64,… URL in the mockup slot and attached as-is to the build handoff.
  • Icon contract — both agents speak lucide. Design emits [icon:<name>] inline in text/html mockups and the icon field in nav. Frontend renders via the Icon component (kebab-case lucide-react names). Zero emoji anywhere. The design-review HTML preview substitutes [icon:<name>] with <img src="https://unpkg.com/lucide-static…/icons/<name>.svg"> for visual fidelity (display only — storage stays in the literal form).
  • Empty/loading/error states must be conditional — the design mockup describes WHAT each state shows; the frontend agent's prompt (mastra/instructions.ts) hard-binds them to visible conditions ({"$state": "/queries/<ds>/page/total", "eq": 0} for empty, isLoading, error) so the empty state doesn't render alongside the populated list.
  • The agent then writes real json-render page specs + app.json navigation — the same runtime contract as before. A page may mix fragments and hand-built primitives.
Resilience note. A Build is the longest, most exposed turn (full context + many tool calls + per-page images). Transient OpenRouter socket drops are retried (maxRetries); if a Build is interrupted, the Rebuild button (§8) re-runs it from the current design as a system action (no chat turn) — wipes the existing page files first so the agent rebuilds cold from the design artifacts instead of refactoring its own prior output.

8 The builder UI /apps/[appId]

One screen: chat on the left, a stage-aware panel on the right, and a pipeline bar across the top that is both a progress stepper and the control surface.

  • Stepper (pipeline-bar.tsx) — Data model · Design · Analyzer · Build. Click any reached step to view its output read-only; a banner offers Re-open to make it the live stage again.
  • Gate actions — Approve data model / Approve design at the gates; a persistent Rebuild at the build stage (always available, independent of the design-changed heuristic); Reset back to the data model. Rebuild is now a system action (POST /rebuild) — wipes the page files and re-runs the frontend agent cold; no chat turn.
  • Models popover — per-app model picker for each role (backend, design, frontend, imageGen). Picking the default clears the override; picking anything else persists. The imageGen slot only shows text→image models; text slots filter those out.
  • Panels — data-model-viewer (entities), design-review (theme card with grouped swatches + light/dark, sitemap, per-page mockup tabs with generate/regenerate + "use for build" + a Fullscreen overlay for the mockup), fragment-analyzer (with Rerun + duration chip), and the AppRuntime live preview.
  • Duration chips everywhere — Theme header, Sitemap header, each mockup mode tab, Analyzer header. Sourced from per-tool stamping (§4). On a "Rerun all", a green banner across the design panel summarises the per-stage breakdown plus the (parallel) wall-clock total.

9 System actions — agent work without a chat turn

Some work is mechanical regeneration, not conversation. These run the relevant agent (or, for image gen, bypass it) server-side and return updated artifacts — no message in the chat.

RouteDoes
POST /mockup/generateGenerate/regenerate ONE mockup representation for ONE page (or all pages, fanned out in parallel). text/html → design agent; image → direct text→image call (bypasses agent). generateMockupRepresentation().
POST /design/rerunRerun a slice of the design from scratch: scope = all · theme · sitemap · mockups. all uses runDesignParallel — theme + sitemap concurrently, then per-page mockups — and returns per-stage timings the UI banners.
POST /analyzerWipe coverage.json, recompute fragment matches from scratch. Returns the new coverage + wall-clock durationMs. (The cache invalidates automatically on sitemap change — this is for fragment-library changes.)
POST /rebuildWipe page files, run the frontend agent cold against the approved design. The rebuild button on the pipeline bar fires this — it was a chat message before, now a system action so prior page specs don't contaminate the prompt.
POST /themeTweaker changes — { preset } picks (pickThemePreset) or token/font/radius edits (applyThemeEdit); rewrites theme.json, powers live preview. Never touches DESIGN.md.
POST /mockupSet which representation is "used for build".
  • Per-tool timing — every design tool stamps durationMs on its artifact at fire time via installDesignTiming(ctx) + stampNextDuration(ctx, set). Works for chat-driven design AND the scoped reruns. Backend/frontend tools have no timing instrumentation.
  • The chat-driven Build still flows through the orchestrator (§2); Rebuild, image gen, and design reruns are system actions.

10 Storage & routes

SQLite — data/builder.db

  • apps · chat_messages · entities · records
  • runs — per-app stage + config (the pipeline state)
  • fragment_sessions · fragment_index (semantic search)

Filesystem — data/<appId>/

  • user-<entity>-<page>.json page specs · app.json (nav + roles)
  • theme.json always present (preset or authored); DESIGN.md only on the create-from-scratch path (§5)
  • design/sitemap.json · design/mockups.json · design/coverage.json
  • builds/<model-slug>/ — auto-snapshot of completed builds per frontend model (A/B archive)

API routes /api/apps/[appId]/*

  • chat (orchestrator) · run (state + per-app models config) · model (entities + theme + sitemap + mockups + pagesStale) · bundle (runtime spec)
  • theme · mockup · mockup/generate · design/rerun · analyzer (GET + POST) · rebuild · datasource
  • Global: /api/models — registry of available models per role + defaults (drives the Models popover).
Mental model: SQLite = run + data, filesystem = design + pages. The agent writes JSON/markdown; the runtime renders it. The pipeline added a Design stage and its artifacts, but never changed who renders the final app.
